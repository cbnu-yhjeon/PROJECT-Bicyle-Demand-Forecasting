# 📊 UCI Bike Sharing 데이터 품질 종합 의견
(분석 대상: `day.csv` · `hour.csv`)

자전거 수요 예측 프로젝트에서 활용하는 **UCI Bike Sharing Dataset**의  
일별(`day.csv`), 시간별(`hour.csv`) 데이터를 대상으로 기본적인 데이터 품질 점검을 수행한 결과를 정리하였다.

---

## 1. 데이터 개요

### 1-1. 파일 구조

- **day.csv (일별 데이터)**
  - 행(row) 수: **731**
  - 열(column) 수: **16**
  - 기간: **2011-01-01 ~ 2012-12-31**
  - 주요 컬럼:
    - `instant` (일련 번호, 1~731 연속)
    - `dteday` (날짜)
    - `season`, `yr`, `mnth`, `holiday`, `weekday`, `workingday`, `weathersit`
    - `temp`, `atemp`, `hum`, `windspeed` (0~1 사이로 정규화된 기상 관련 변수)
    - `casual`, `registered`, `cnt` (이용자 수)

- **hour.csv (시간별 데이터)**
  - 행(row) 수: **17,379**
  - 열(column) 수: **17**
  - 기간: **2011-01-01 ~ 2012-12-31**
  - 주요 컬럼:
    - `instant` (일련 번호, 1~17379 연속)
    - `dteday`, `hr` (날짜, 시각)
    - 나머지 구조는 day.csv와 유사 (`season`, `yr`, `mnth` 등)
    - `casual`, `registered`, `cnt` (시간 단위 이용자 수)

---

## 2. 결측치·중복·기본 품질

### 2-1. 결측치(Missing Value)

- `day.csv`, `hour.csv` **모든 컬럼에서 결측치 0개**
- `isna().sum()` 기준, 전 컬럼 결측 없음 → **결측치 품질 매우 우수**

### 2-2. 중복(Duplicate Row)

- `day.csv` 중복 행 수: **0**
- `hour.csv` 중복 행 수: **0**
- `instant` 값도 각각
  - day: 1 ~ 731
  - hour: 1 ~ 17379  
  **연속된 정수 범위로 잘 구성**되어 있음 → 인덱스/레코드 관리 양호

---

## 3. 값의 범위 및 유효성

### 3-1. 주요 연속형 변수 범위

- **일별(day.csv)**
  - `temp`: 0.059 ~ 0.862
  - `atemp`: 0.079 ~ 0.841
  - `hum`: 0.000 ~ 0.973
  - `windspeed`: 0.022 ~ 0.507
  - `cnt`: 22 ~ 8,714
  - `casual`: 2 ~ 3,410
  - `registered`: 20 ~ 6,946

- **시간별(hour.csv)**
  - `temp`: 0.02 ~ 1.00
  - `atemp`: 0.00 ~ 1.00
  - `hum`: 0.00 ~ 1.00
  - `windspeed`: 0.00 ~ 0.851
  - `cnt`: 1 ~ 977
  - `casual`: 0 ~ 367
  - `registered`: 0 ~ 886

👉 **해석**
- 기상 관련 변수(`temp`, `atemp`, `hum`, `windspeed`)는 **0~1 범위로 잘 정규화**되어 있음.
- 수요 관련 변수(`cnt`, `casual`, `registered`)는 음수가 존재하지 않고,  
  일/시간 스케일에 비해 상식적인 범위에 위치.

### 3-2. 범주형 변수 유효성

- `season`: 1~4
- `yr`: 0, 1 (2011, 2012)
- `mnth`: 1~12
- `weekday`: 0~6
- `weathersit`:
  - 일별(day): 1~3
  - 시간별(hour): 1~4 (4는 극단적 악천후, 매우 희귀하게 등장)

👉 **해석**
- 라벨 값 범위가 정의된 카테고리 내에 있음.
- 시간별 데이터에서만 `weathersit = 4`가 등장하는데,  
  일별 데이터엔 1~3만 존재 → **집계 과정에서 극악한 날씨가 묶이거나 제외된 것으로 추정**  
  (이상치는 아니지만, 두 데이터셋 비교 시 인지해야 할 차이)

---

## 4. 내부 일관성(Consistency)

### 4-1. `cnt = casual + registered` 관계

- `day.csv`, `hour.csv` 전 행에서:
  - **`cnt == casual + registered` 조건이 100% 성립**
- 이용자 수 집계에 **논리적 일관성 매우 우수**

### 4-2. 일별 vs 시간별 집계 일치 여부

- `hour.csv`를 `dteday` 기준으로 집계(`sum`) 후,
  - `casual`, `registered`, `cnt`를 day.csv와 비교한 결과:
    - 각 날짜별로 **일별 값 == 시간별 합계** 가 정확히 일치
- 즉, `day.csv`는 `hour.csv`의 단순 합계 버전으로 볼 수 있어  
  **두 파일 간 수요 데이터 일관성이 매우 높음**

---

## 5. 잠재적인 이슈 및 해석상 주의점

### 5-1. 습도·풍속의 0 값

- `hum == 0`:
  - day: 1건
  - hour: 22건
- `windspeed == 0`:
  - day: 0건
  - hour: 약 2,180건(전체의 상당 비율)

👉 **해석**
- 풍속 0은 “무풍 상태”로도 해석 가능하나,  
  측정 장비의 민감도/기록 방식에 따라 **하한값 클리핑 혹은 결측 대체일 가능성**도 있음.
- 모형 설계 시:
  - 완전한 0을 “측정 불능/결측 의심값”으로 보고
  - 소량의 무작위 노이즈를 추가하거나,
  - 별도의 indicator feature(`is_windspeed_zero`)로 처리하는 전략도 고려 가능.

### 5-2. 정규화 스케일 문제

- `temp`, `atemp`, `hum`, `windspeed` 등이 이미 **0~1 스케일로 정규화**되어 있음.
- 이 데이터(UCI)와 **서울시 실제 따릉이 데이터(원 단위 값, ℃, m/s 등)**를 통합할 경우:
  - 스케일이 전혀 다르므로 그대로 합치면 **모델 입력 스케일이 섞이는 문제 발생**
  - 반드시:
    1. 원 데이터 단위로 되돌리는 과정(가능하면) 또는
    2. 통합 후 전체 피처에 대해 다시 일괄 스케일링(StandardScaler/MinMax 등)  
    을 수행해야 함.

### 5-3. 기간 범위와 대표성

- 전체 기간은 **2011~2012년 2년치**로 제한되어 있음.
- 최근 서울시 따릉이 데이터(2020~2025)와는:
  - 도시 교통 패턴, 날씨, 인구 구조, 자전거 인프라 등이 상당히 다를 수 있음.
- 따라서 이 UCI 데이터는:
  - **모델 구조·특성 탐색, 피처 엔지니어링 실험용 “레퍼런스/전이 학습용 데이터”**로 적합하고,
  - **서울시 실제 수요 예측 모델의 성능을 직접적으로 대변하는 데이터로 보기에는 한계**가 있음.

### 5-4. 날씨 정보의 간소화

- `weathersit`은 1~4 구간으로 날씨를 매우 단순화해서 표현:
  - 예: 맑음 / 흐림 / 가벼운 비 / 심한 비, 눈 등
- 실제 서울 기상청 데이터(ASOS)의:
  - 강수량(mm), 적설량, 시정, 기압 등의 세밀한 정보를 사용하려면
  - **기상 정보의 해상도가 UCI보다 훨씬 높아질 것** → 피처 구조가 달라짐

## 6. 종합 평가

### 6-1. 장점

1. **결측치 없음**
   - 전 컬럼에 결측이 없고, 기본 전처리가 잘 되어 있음.

2. **중복 없음 & 인덱스 정합성 우수**
   - `instant`가 연속적이고, 중복 행 없음.

3. **일별-시간별 완벽한 수요 집계 일치**
   - `day.csv`와 `hour.csv` 사이에 수요 데이터(`cnt`, `casual`, `registered`)가  
     날짜 기준으로 100% 일치 → **집계 수준 별 분석이 매우 안정적**.

4. **기본적인 값 범위가 현실적**
   - 이용자 수가 음수가 없고, 과도하게 비현실적인 값도 없음.
   - 기상 변수는 0~1로 전처리되어 있어 모델링에 바로 사용하기 좋음.

### 6-2. 한계 및 주의점

1. **기간이 2011~2012로 제한**
   - 최신 서울시 따릉이 수요(2020~2025)와 환경이 다르므로,
   - **모델의 일반화/현실 반영 측면에서 직접 사용에는 한계**가 있음.

2. **정규화 스케일 불일치**
   - UCI 데이터는 정규화, 서울 열린데이터+기상청 데이터는 원 단위 →  
     통합 시 반드시 재스케일링 필요.

3. **일부 0값(특히 풍속)의 해석**
   - 풍속 0이 다수 존재해,
   - **측정 하한/기록 처리 방식**까지 고려한 해석이 필요.

---

## 7. 프로젝트 관점의 결론

- **데이터 품질 자체(결측·중복·내부 일관성)는 매우 우수한 편**이며,
  - 실습/연구용 벤치마크 데이터로서의 완성도는 높다.
- 다만, 본 프로젝트의 목표가 **“서울시 2020~2025 따릉이 수요 예측”**이므로,
  - UCI 데이터는
    - 피처 구조 설계
    - 모델 아키텍처 실험
    - 전이 학습용(Pretrain → 서울 데이터 Fine-tune)  
    정도로 활용하는 것이 적절하다.
- 실제 서비스 수준의 예측 모델은:
  - **서울 열린데이터(2020~2025.06) + 기상자료개방포털(ASOS)**을 기반으로 구축하고,
  - UCI 데이터는 **“참고용/보조 데이터셋”으로 위치를 잡는 것이 바람직**하다.
